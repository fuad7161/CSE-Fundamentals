# Computer Software

Everything about software — system vs application, programming languages, translators, and the software ecosystem.

## 9.1 — Software vs Hardware — Definition & Relationship

### Definitions
- **Hardware**: The physical, tangible components of a computer system — CPU, memory, hard drive, motherboard, peripherals. You can touch it.
- **Software**: A collection of instructions and data that tells hardware what to do. It is intangible — a set of programs, procedures, and documentation.

### Relationship
- Hardware and software are **interdependent**.
- Hardware provides the physical capability to process and store data.
- Software provides the instructions that direct the hardware to perform specific tasks.
- Without software, hardware is an inert collection of circuits (no OS to boot, no programs to run).
- Without hardware, software has no physical medium to execute on.

### Layers of Abstraction
1. **Hardware** — bare metal (CPU, RAM, disk)
2. **Operating System** — manages hardware and provides services to applications
3. **Application Software** — user-facing programs that run on top of the OS

## 9.2 — System Software

**System software** is a type of computer program designed to run a computer's hardware and application programs. It acts as an intermediary between hardware and user applications.

### Operating System (OS)
- The primary system software that manages all hardware and software resources.
- Provides a **user interface** (CLI or GUI) for interaction.
- Handles process scheduling, memory allocation, file management, I/O operations, and security.
- Examples: **Windows** (11, 10, Server), **Linux** (Ubuntu, Debian, Fedora, RHEL), **macOS**, **Android**, **iOS**.

### Device Drivers
- Specialized programs that allow the operating system to communicate with specific hardware devices.
- Each hardware component (graphics card, network adapter, printer, mouse) needs a driver.
- Drivers translate generic OS commands into device-specific instructions.
- Often provided by the hardware manufacturer and updated to fix bugs or improve performance.

### Utility Programs
- System software designed to help analyze, configure, optimize, and maintain the computer.
- Common utilities:
  - **Disk Cleanup**: Removes temporary files, empties recycle bin.
  - **Antivirus / Anti-malware**: Scans and removes malicious software.
  - **Backup Software**: Creates copies of data for recovery.
  - **File Compression**: WinRAR, 7-Zip, gzip — reduce file size.
  - **Disk Defragmentation**: Reorganizes fragmented data on HDDs for faster access.
  - **System Monitor**: Task Manager, Resource Monitor, Performance Monitor.

## 9.3 — Application Software

**Application software** (also called "apps" or "programs") is designed to help users perform specific tasks. It runs on top of system software.

### General-Purpose Application Software
- Designed for a broad audience to perform common tasks.
- Examples:
  - **Word Processor**: Microsoft Word, Google Docs, LibreOffice Writer.
  - **Spreadsheet**: Microsoft Excel, Google Sheets.
  - **Web Browser**: Google Chrome, Mozilla Firefox, Microsoft Edge.
  - **Email Client**: Outlook, Thunderbird, Gmail (web app).
- Used by anyone — students, office workers, home users.

### Specific-Purpose Application Software
- Tailored for a particular industry, organization, or specialized task.
- Examples:
  - **Hospital Management System**: Manages patient records, appointments, billing.
  - **Airline Reservation System**: Handles flight bookings, seat assignments, check-in.
  - **Payroll Software**: Calculates salaries, taxes, deductions for employees.
  - **CAD (Computer-Aided Design)**: AutoCAD, SolidWorks — used by engineers and architects.
  - **Video Editing**: Adobe Premiere Pro, DaVinci Resolve, Final Cut Pro.
- Usually not useful to a general user — built for specific professional needs.

| Aspect | General Purpose | Specific Purpose |
|--------|-----------------|------------------|
| Audience | Broad (everyone) | Niche (specialists) |
| Cost | Low to moderate | Often high (custom development) |
| Customization | Limited | Highly customizable |
| Examples | Word, Excel, Chrome | AutoCAD, ERP systems |

## 9.4 — Programming Languages

A **programming language** is a formal set of instructions used to communicate with a computer and develop software.

### Generations of Programming Languages

#### 1st Generation — Machine Language
- Binary code (0s and 1s) executed directly by the CPU.
- **Hardware-dependent** — each CPU architecture has its own machine language.
- Extremely tedious and error-prone for humans.

#### 2nd Generation — Assembly Language
- Uses **mnemonics** (human-readable abbreviations) instead of binary.
  - Example: `MOV AX, 5` instead of `10111000 00000101 00000000`.
- Requires an **assembler** to translate to machine code.
- Still low-level and architecture-specific.

#### 3rd Generation — High-Level Language
- English-like syntax, hardware-independent.
- Uses **compilers or interpreters** to translate to machine code.
- Examples: **C, C++, Java, Python, C#, JavaScript, Ruby, Go, Rust**.
- Features: variables, loops, functions, data structures, abstraction.

#### 4th Generation (4GL) — Declarative / Domain-Specific
- Less code needed than 3GL — focuses on *what* to do, not *how*.
- Examples: **SQL** (database queries), **MATLAB** (numerical computing), **R** (statistics).
- Often used for database access, report generation, data analysis.

#### 5th Generation (5GL) — AI / Constraint-Based
- Programs describe problems and constraints; the system finds the solution.
- Examples: **Prolog** (logic programming), **Lisp** (symbolic AI), **CLIPS** (expert systems).
- Used in artificial intelligence, natural language processing, expert systems.

## 9.5 — Language Translators

Language translators convert source code written in one language into another form (usually machine code).

### Compiler
- Translates the **entire source code** at once into an executable machine code file.
- Produces a standalone executable (`.exe`, `.elf`) that can be run independently.
- **Faster execution** because translation is done beforehand.
- **Slower debugging** because you must recompile after every change.
- Examples: GCC (C/C++), javac (Java — compiles to bytecode), rustc (Rust).

### Interpreter
- Translates and executes source code **line by line**.
- No separate executable is created — the interpreter must be present at runtime.
- **Slower execution** because translation happens during runtime.
- **Easier debugging** because you can stop and inspect at any line.
- Examples: Python (CPython), Ruby, JavaScript (in browsers), Bash.

### Assembler
- Translates **assembly language** (mnemonics) into machine code.
- One-to-one mapping: one assembly instruction ≈ one machine code instruction.
- Used in low-level programming, compilers, and operating system development.

### Hybrid Approach (Java / .NET)
- **Java**: Source code is **compiled** by `javac` into platform-independent **bytecode** (`.class` files).
- Bytecode is then **interpreted** (or JIT-compiled) by the **Java Virtual Machine (JVM)**.
- This gives portability ("write once, run anywhere") while maintaining reasonable performance.
- **.NET (C#)**: Similar — compiled to **CIL (Common Intermediate Language)**, then JIT-compiled by the **CLR (Common Language Runtime)**.

| Translator | Approach | Output | Speed of Execution | Example |
|-----------|----------|--------|-------------------|---------|
| Compiler | Entire source at once | Standalone executable | Fast | GCC, javac |
| Interpreter | Line by line | No standalone file | Slower | Python, Ruby |
| Assembler | Assembly → machine | Executable | Very fast | NASM, MASM |
| Hybrid | Compile → intermediate → JIT | Bytecode + runtime | Good | Java (JVM), C# (.NET) |

## 9.6 — Operating System — Functions & Types

### Functions of an Operating System

1. **Process Management**
   - Creates, schedules, and terminates processes.
   - Manages multitasking (CPU time-sharing between processes).
   - Synchronization and inter-process communication (IPC).

2. **Memory Management**
   - Allocates and deallocates memory to running programs.
   - Handles virtual memory (paging/swapping to disk).
   - Protects memory spaces between processes.

3. **File System Management**
   - Organizes data into files and directories (folders).
   - Controls read/write permissions.
   - Supports different file systems: NTFS, FAT32, ext4, APFS.

4. **I/O Device Management**
   - Communicates with hardware via device drivers.
   - Buffers and spools data to/from devices.
   - Handles interrupts from hardware.

5. **Security & Access Control**
   - User authentication (login, passwords, biometrics).
   - File permissions (read, write, execute).
   - Protection against unauthorized access.

6. **User Interface**
   - **CLI (Command-Line Interface)**: Shell, terminal (Linux Bash, Windows cmd/powershell).
   - **GUI (Graphical User Interface)**: Desktop, icons, windows (Windows Explorer, macOS Finder).

### Types of Operating Systems

| Type | Description | Examples |
|------|-------------|----------|
| **Batch OS** | Jobs are grouped together and executed without user interaction | IBM OS/360 |
| **Multiprogramming / Multitasking** | Multiple programs run simultaneously by switching CPU between them | Windows, Linux, macOS |
| **Multiuser OS** | Multiple users can access the same system simultaneously (via terminals) | Unix, Linux, Windows Server |
| **Real-Time OS (RTOS)** | Guarantees response within a strict time deadline | FreeRTOS, VxWorks, QNX |
| **Network OS** | Manages network resources; runs on servers | Windows Server, Linux, Novell NetWare |
| **Embedded OS** | Runs on devices with limited resources (smart appliances, IoT) | Embedded Linux, RTOS, Android Things |
| **Distributed OS** | Manages a group of independent computers as a single system | Plan 9, Amoeba, Google Fuchsia |

## 9.8 — Open Source vs Proprietary Software

### Open Source Software (OSS)
- **Source code** is freely available for anyone to view, modify, and distribute.
- Developed collaboratively by communities or organizations following an open-source license.
- **Licenses**: GNU GPL, MIT, Apache, BSD.
- **Examples**:
  - **Linux**: Open-source kernel, basis of Ubuntu, Debian, Fedora.
  - **Mozilla Firefox**: Open-source web browser.
  - **LibreOffice**: Open-source office suite (Word, Excel, PowerPoint alternatives).
  - **VLC Media Player**: Open-source media player.
  - **GIMP**: Open-source image editor (Photoshop alternative).

### Proprietary Software
- **Source code** is owned by an individual or company; it is **not** publicly available.
- Users purchase a license to use the software but cannot modify or redistribute it.
- The owner controls development, updates, and pricing.
- **Examples**:
  - **Windows**: Microsoft owns and licenses the code.
  - **Microsoft Office**: Proprietary office suite.
  - **Adobe Photoshop / Premiere Pro**: Proprietary creative tools.
  - **macOS / iOS**: Apple's proprietary operating systems.

### Comparison

| Aspect | Open Source | Proprietary |
|--------|-------------|-------------|
| Source code | Publicly available | Private, closed |
| Cost | Usually free | Paid license |
| Customization | Full freedom | Limited by vendor |
| Support | Community forums, paid options possible | Vendor support (often paid) |
| Security | Transparent, peer-reviewed | Hidden from public scrutiny |
| Updates | Community-driven | Vendor-controlled |

### Freeware vs Open Source
- **Freeware** is proprietary software distributed at no cost, but the source code is not available (e.g., Skype, Adobe Acrobat Reader).
- **Open source** is free *and* provides source code access.

## 9.9 — Firmware vs Software

### Firmware
- A specialized type of software stored on **non-volatile memory** (ROM, EPROM, EEPROM, or Flash) on a hardware device.
- Provides low-level control for the hardware.
- Is **permanently** stored and is not easily modified by the end user (though modern firmware can be updated via "flashing").
- **Examples**:
  - **BIOS / UEFI**: Firmware on the motherboard that initializes hardware and boots the OS.
  - **Router Firmware**: DD-WRT, OpenWrt, or the vendor's stock firmware.
  - **Hard drive / SSD firmware**: Controls drive operations, error correction, wear leveling.
  - **Embedded device firmware**: Smart TV, microwave, car ECU, IoT sensor.
- **Updating firmware** is riskier than updating software — a failed update can "brick" the device.

### Software (as distinguished from firmware)
- Stored on **secondary storage** (HDD, SSD, USB drive) and loaded into RAM when needed.
- Designed to be **easily modified, updated, or replaced** by the user.
- Includes the operating system, applications, drivers, utilities.
- Updates are routine and generally safe.

### Key Differences

| Aspect | Firmware | Software |
|--------|----------|----------|
| Storage medium | ROM, Flash, EPROM (non-volatile) | HDD, SSD, optical disc |
| Update frequency | Rare (sometimes never) | Frequent |
| Update difficulty | Requires special tools/procedures | Simple (install/uninstall) |
| Purpose | Control hardware at low level | Perform user-oriented tasks |
| Volatility | Permanent (survives power-off) | Can be deleted/reinstalled |
| Failure consequence | Can "brick" the device | Typically reinstallable without hardware damage |
